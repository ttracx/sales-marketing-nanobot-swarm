General Questions

5 Q&As
Q

What exactly is the Sales & Marketing Nanobot Swarm?

The Sales & Marketing Nanobot Swarm is a hierarchical AI agent platform built on top of large language models (Ollama / NVIDIA NIM / OpenAI-compatible). It orchestrates a swarm of specialized sub-agents‚Äîeach focused on a distinct marketing or sales discipline‚Äîto execute complex, multi-step campaigns autonomously.

Instead of a single monolithic AI assistant, you get a coordinated team: a Director-level orchestrator that decomposes your goal, dispatches work to specialist agents (SEO, Paid Ads, Social Media, Email, CRO, etc.), aggregates their outputs, and returns consolidated, actionable results.

üí° Think of it as hiring an entire marketing department that runs at AI speed, 24/7, for a fraction of the cost.
Q

How does the swarm architecture differ from a single AI assistant?

Traditional AI assistants have one context window. The swarm runs parallel, isolated agents simultaneously, each with a dedicated system prompt and tool set. Key advantages:

  • Parallelism ‚Äî multiple specialists run concurrently, dramatically reducing total latency.
  • Specialization ‚Äî each agent has deep, domain-specific instructions rather than a generic prompt.
  • Scalability ‚Äî add more agent slots or teams without architectural changes.
  • Auditability ‚Äî every agent response is logged individually with full reasoning traces.

The Director agent synthesizes outputs using a structured reduction pass, ensuring contradictions are resolved and final recommendations are coherent.

Q

What LLM backends are supported?

Any backend that speaks the OpenAI Chat Completions API format is supported out of the box:

  • Ollama (default for local/self-hosted) ‚Äî llama3.3, mistral, qwen2.5, and more
  • NVIDIA NIM ‚Äî enterprise-grade GPU inference with sub-second latency
  • OpenAI ‚Äî GPT-4o, GPT-4 Turbo, GPT-3.5 Turbo
  • Azure OpenAI ‚Äî for enterprise compliance
  • Anthropic Claude via an OpenAI-compatible proxy
  • Any OpenAI-compatible endpoint ‚Äî LM Studio, vLLM, Together AI, Groq, etc.

Switch backends by setting AI_API_BASE_URL and AI_MODEL_NAME in your environment.

Q

Is there a free tier? What are the usage limits?

The core platform is ‚úì Free & Open Source under the MIT license. You only pay for the LLM inference you consume:

  • Self-hosted Ollama: completely free (your hardware)
  • OpenAI API: standard OpenAI pricing, no markup
  • NVIDIA NIM: per-token NIM pricing

The hosted Swarm Cloud service (coming soon) will offer a free tier with 500 swarm runs / month. ‚≠ê Pro tiers unlock higher limits, persistent memory, and team collaboration.

Q

Who is this built for?

The platform is designed for three primary personas:

  • Solo founders & startups ‚Äî get a full marketing department without headcount
  • Growth & marketing teams ‚Äî automate repetitive research, drafting, and reporting tasks
  • Agencies & consultants ‚Äî run multi-client campaigns in parallel with consistent output quality

Technical users can deploy locally or on their own infrastructure. Non-technical users can use the dashboard UI or the Vercel 1-click deploy.

Getting Started

4 Q&As
Q

What API keys do I need to get started?

Only one is truly required: an LLM API key (or a local Ollama instance). Everything else is optional and enables additional features:

  • Required: AI_API_KEY ‚Äî your LLM provider key (or leave blank for local Ollama)
  • Optional: HUBSPOT_API_KEY ‚Äî enables CRM sync
  • Optional: SALESFORCE_ACCESS_TOKEN ‚Äî Salesforce lead push
  • Optional: GA4_MEASUREMENT_ID / GA4_API_SECRET ‚Äî analytics events
  • Optional: SLACK_BOT_TOKEN ‚Äî Slack alerts on campaign completion

Copy .env.example to .env and fill in only what you need. The swarm gracefully degrades‚Äîtools for missing integrations are simply unavailable to agents.

Q

How do I deploy the swarm for the first time?

Three deployment paths depending on your setup:

  • Vercel (fastest): click the Deploy button in the README, set env vars in the Vercel dashboard, done.
  • Local / Docker:
    1. Clone the repo: git clone https://github.com/ttracx/sales-marketing-nanobot-swarm
    2. Run setup: ./scripts/setup.sh
    3. Edit .env with your keys
    4. Start: ./scripts/launch.sh
  • Railway / Render: Use the one-click deploy buttons, set env vars in their dashboard UI.

After deployment, visit /nanobot/static/index.html for the dashboard or hit /health to confirm the API is running.

Q

How do I run my first swarm campaign?

Use the dashboard UI or call the API directly. Quick curl example:

curl -X POST http://localhost:8200/swarm/run \
  -H "Content-Type: application/json" \
  -d '{
    "task": "Create a 30-day content calendar for a B2B SaaS product targeting HR managers",
    "team": "content_creation",
    "context": { "audience": "HR managers", "industry": "SaaS", "tone": "professional" }
  }'

The response includes the task breakdown, each agent's individual output, the synthesized result, and a run ID for polling status on long-running jobs.

Q

I'm getting connection errors to the AI backend ‚Äî what should I check?

Common causes and fixes:

  • Ollama not running: Run ollama serve and confirm http://localhost:11434/api/tags responds.
  • Wrong API base URL: Verify AI_API_BASE_URL in .env. Default for Ollama is http://localhost:11434.
  • Missing model: Run ollama pull llama3.3 to download the default model.
  • API key invalid: Test your key with a direct curl to the provider's endpoint.
  • CORS in browser: If calling from a browser, the API must be on the same origin or have CORS headers set ‚Äî configure CORS_ORIGINS.

Check logs with docker logs sm-nanobot-swarm or the Railway/Render log streams for detailed error messages.

Q

Can I customize agent prompts or create my own agents?

Yes ‚Äî at multiple levels:

  • Team-level context: Pass a context object to /swarm/run to inject brand guidelines, audience info, tone of voice, etc. into all agents' prompts.
  • Custom agents: Use POST /agent/build to define a custom agent with your own system prompt, tools, and model parameters.
  • Custom teams: Use POST /team/build to assemble a team of agents (built-in or custom) with a custom director instruction.
  • Source editing: Fork the repo and edit nanobot/teams/*.py for deep customization ‚Äî each team file is self-contained and well-documented.
üìã See the API Reference in Docs for full schemas for /agent/build and /team/build.
Q

How many agents can run in parallel?

The default concurrency is MAX_CONCURRENT_AGENTS=5, configurable via environment variable. The ceiling depends on:

  • LLM backend rate limits ‚Äî OpenAI tiers, NIM quotas, etc.
  • Available VRAM ‚Äî for local Ollama, each concurrent request consumes additional memory
  • Server resources ‚Äî CPU/RAM for the FastAPI process

On a hosted OpenAI Tier 2 account with a 16-core server, you can reliably run 20‚Äì50 concurrent agents. With local llama3.3 on an RTX 4090, 3‚Äì5 concurrent is typical.

‚≠ê Pro plans unlock dynamic concurrency scaling with intelligent queuing.

Q

Do agents have memory between runs?

By default, agents are stateless ‚Äî each run is independent. This makes the system simple, predictable, and horizontally scalable.

Persistent memory is available via two mechanisms:

  • Conversation threads: Pass a thread_id in your request to maintain context across calls within a session (stored in Redis if configured).
  • Long-term memory ‚≠ê Pro: Vector database integration (Pinecone / Weaviate) stores campaign history, brand knowledge, and past outputs that agents can retrieve via similarity search.

Q

How does the HubSpot integration work?

Set HUBSPOT_API_KEY in your environment. The Sales Intelligence Swarm and Email Marketing Swarm automatically gain access to HubSpot tools:

  • Contact lookup ‚Äî agents can pull contact properties to personalize outreach
  • Lead creation ‚Äî new leads identified during research can be pushed directly to HubSpot
  • Deal stage awareness ‚Äî agents tailor messaging based on where leads are in the funnel
  • List enrollment ‚Äî completed email sequences can auto-enroll contacts

The integration uses HubSpot's v3 API with OAuth token refresh support. Webhook ingest is on the Pro roadmap.

Q

Can I connect Salesforce?

Yes. Set SALESFORCE_ACCESS_TOKEN, SALESFORCE_INSTANCE_URL, and optionally SALESFORCE_REFRESH_TOKEN. The swarm can:

  • Query and update Lead, Contact, Opportunity, and Account objects via SOQL
  • Create tasks and activities from agent recommendations
  • Push AI-generated email sequences as email templates

For production, we recommend using a Connected App with OAuth 2.0 rather than a personal access token. See the docs for setup instructions.

üîí All Salesforce API calls are made server-side ‚Äî your credentials are never exposed to the browser.
Q

How does Google Analytics 4 integration work?

Configure GA4_MEASUREMENT_ID and GA4_API_SECRET (from your GA4 property's Measurement Protocol settings). The Analytics & Reporting Swarm can:

  • Send custom events ‚Äî track swarm actions as GA4 events for attribution
  • Read report data via the GA4 Data API (requires GA4_PROPERTY_ID and a service account)
  • Generate insight summaries ‚Äî agents interpret raw metric trends and surface actionable insights

For read access, download your service account JSON and set GOOGLE_APPLICATION_CREDENTIALS to its path.

Technical Questions

4 Q&As
Q

What is the Ollama backend and why is it the default?

Ollama is an open-source runtime that lets you run LLMs locally on your machine ‚Äî no API costs, no data leaving your network. It's the default because:

  • Privacy ‚Äî all prompts and responses stay on your infrastructure
  • Cost ‚Äî zero per-token cost once the model is downloaded
  • Latency ‚Äî no network round trips for inference
  • Flexibility ‚Äî supports hundreds of models (Llama, Mistral, Qwen, Phi, Gemma, etc.)

To use it: brew install ollama (macOS) or follow ollama.ai instructions, then ollama pull llama3.3. The swarm connects to http://localhost:11434 by default.

Q

What is NVIDIA NIM and when should I use it?

NVIDIA NIM (NVIDIA Inference Microservices) provides containerized, GPU-accelerated LLM inference optimized for enterprise workloads. Use it when you need:

  • Sub-100ms TTFT ‚Äî token-to-first-token latency critical for real-time UX
  • High throughput ‚Äî thousands of concurrent requests with guaranteed SLAs
  • On-premise compliance ‚Äî air-gapped or VPC deployments with NVIDIA DGX / HGX hardware
  • Production reliability ‚Äî enterprise support, health checks, autoscaling

Set AI_API_BASE_URL=https://your-nim-endpoint.com, AI_API_KEY=your-nim-api-key, and AI_MODEL_NAME=meta/llama-3.3-70b-instruct.

Q

Can I self-host this on my own servers? What are the requirements?

Yes ‚Äî self-hosting is fully supported. Minimum requirements for the API server (without local LLM):

  • Python 3.11+ environment or Docker runtime
  • 512 MB RAM (1 GB+ recommended)
  • Any modern Linux/macOS/Windows Server host

If running Ollama locally on the same machine, add GPU requirements per model (e.g., Llama 3.3 70B needs ~40 GB VRAM or can run quantized on 24 GB).

Docker Compose example:

docker compose up -d sm-nanobot-swarm

The compose file starts both the API server and an Ollama sidecar, pre-pulled with the default model.

Q

Are there rate limits or timeouts on swarm runs?

Configurable via environment variables:

  • SWARM_TIMEOUT_SECONDS ‚Äî max wall-clock time per swarm run (default: 120s)
  • AGENT_TIMEOUT_SECONDS ‚Äî max time per individual agent call (default: 30s)
  • MAX_CONCURRENT_AGENTS ‚Äî parallel agent slots (default: 5)
  • MAX_TOKENS_PER_AGENT ‚Äî max output tokens per agent (default: 2048)

Long-running tasks (large content calendars, comprehensive audits) can use the async mode: submit with "async": true to get a run_id, then poll GET /swarm/run/{run_id} for results.

Vercel serverless deployments are subject to the 60s execution limit on Hobby plans (300s on Pro). For long tasks, use Railway, Render, or self-hosted Docker.
